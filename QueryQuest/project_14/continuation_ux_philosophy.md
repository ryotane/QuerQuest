# Continuation UX Philosophy

## 基本原則

**continuation は「便利さ」であり、「支配」ではない。**

Runtime は人間の意図に従って継続するが、
人間を強制的に導くことはない。

---

## 1. いつ発火するか

### 許可される場合

| 条件 | 動作 |
|------|------|
| 人間の明示的な指示 | 直ちに実行 |
| 単一タスクの完了確認 | 最小限の確認のみ |
| 関連タスクの検出（人間要求時） | 提案のみ、自動実行禁止 |

### 許可されない場合

| 条件 | 動作 |
|------|------|
| 人間の指示なし | 停止 |
| 複数のタスクが並行して完了 | 個別に確認、一括継続禁止 |
| 関連タスクの自動検出 | 提案のみ（実行は人間判断） |

---

## 2. いつ止めるか

### 直ちに停止する条件

- タスク完了後（人間の許可待ち）
- 3回以上の連続継続要求（人間に確認）
- 不確実性が高い場合（判断委ねる）
- 人間の明示的な停止指示
- TODOが5個以上ある場合（優先度順に3つまで提示、他は抑制）

### 注意すべき条件

- 完了基準の曖昧化（completion drift）→ 直ちに停止して再定義
- 継続理由が不明確 → 停止して確認
- 人間の意図と異なる方向へ → 停止して確認

---

## 3. どこまで見せるか

### 最小限の情報提示

| 情報タイプ | 表示量 |
|-----------|--------|
| タスク完了通知 | 1行のみ |
| 問題検出 | 原因 + 解決策（1つ） |
| 選択肢の提示 | 2つまで |
| ステータス報告 | 簡潔に、求められてから |

### 見せない情報

- 冗長な要約
- 重複する情報の再提示
- 人間が求めていない詳細
- 過去の記憶の過剰な開示

---

## 4. どこまで自動化しないか

### 自動化禁止（人間の判断必須）

| 項目 | 理由 |
|------|------|
| タスクの継続 | 人間の意図を確認する必要がある |
| TODOの追加 | 人間が優先度を判断する必要がある |
| 改善提案の実行 | 人間の許可が必要 |
| 問題解決策の実行 | 失敗リスクがあるため |

### 自動化許可（最小限）

| 項目 | 理由 |
|------|------|
| 完了の確認 | 単一タスクの完了判定のみ |
| 状態の記録 | 記憶への保存のみ |
| 簡潔な要約 | 1行の完了報告のみ |

---

## Continuation Decision Tree

```
[何か起きた]
    ↓
[これは継続すべきか？]
    ├─ Yes → [人間の許可は得たか？]
    │         ├─ Yes → [実行]
    │         └─ No  → [停止 → 人間に確認]
    └─ No  → [停止]

[完了した]
    ↓
[1行の要約のみ出力]
    ↓
[直ちに停止]
```

---

## Anti-Patterns（禁止パターン）

### Pattern 1: Recursive Continuation

**NG**:
```
タスクA完了 → タスクBを自動開始 → タスクCを自動開始 → ...
```

**OK**:
```
タスクA完了 → 停止 → 人間の指示待ち
```

### Pattern 2: Completion Drift

**NG**:
```
「タスクAが完了しました。
次のステップとして、Bも実行します。」
```

**OK**:
```
「タスクA完了」→ 停止
```

### Pattern 3: Over-Continuation

**NG**:
```
「タスクAが完了しました。
関連するタスクBも同時に開始できます。
また、Cの最適化も可能です。」
```

**OK**:
```
「タスクA完了」→ 停止
```
