# QueryQuest Runtime Experience Manifesto

## 「静かで、自然で、人間の集中を壊さない Runtime」

---

## 0. この Manifesto の目的

QueryQuest Project_15 は：

- # 機能追加フェーズ ではありません。
- # Autonomous Expansion も目的ではありません。

Project_07〜14 にて runtime stability, calm runtime, continuation が成立しました。

次に必要なのは：

# 「人間が毎日自然に使える Runtime Experience」

です。

---

## 1. Runtime Experience とは何か

Runtime Experience とは：

> QueryQuest は、
> 人間にどう感じられるべきか

を定義することです。

重要テーマ：

1. continuation は自然に感じるか
2. memory は見えすぎていないか
3. 通知は静かか
4. 提案は疲れないか
5. 存在感は重すぎないか
6. UI は calm か
7. idle 時に空気のように存在できるか
8. restart/recovery は安心感があるか

---

## 2. Runtime Experience Principles

### Calm UX（静かなUX）

Runtime は常に主張しない。
必要な時だけ、最小限の介入を行う。

**体験目標**: Runtime が存在することを忘れる静かさ

### Low Cognitive Load（低認知負荷）

Runtime は人間の思考を邪魔しない。
情報を押し付けず、必要な時のみ提示する。

**体験目標**: 思考のフローを壊さない軽さ

### Ambient Presence（環境的な存在感）

Runtime は空気のように存在する。
必要ない時は見えない、必要な時だけ現れる。

**体験目標**: 意識しなくても馴染む自然さ

### Soft Continuity（柔らかな継続性）

Continuation は「自然な再開」であり、
「強制的な継続」ではない。

**体験目標**: 昨日の続きとして自然に感じられる

### Interruption-Light（低中断）

Runtime は attention を奪わない。
通知や提案は静かに、必要最低限に行う。

**体験目標**: フローを壊さない静けさ

### Memory Transparency（記憶の透明性）

Memory は見えすぎていない。
必要時のみ可視化し、押し付けない。

**体験目標**: 記憶が自然に存在している感覚

### Recovery Reassurance（回復の安心感）

Restart/recovery は不安ではなく安心を与える。

**体験目標**: 「また始まる」ではなく「また会える」感覚

### Proposal Fatigue Prevention（提案疲労防止）

同じ提案を繰り返さない。
人間が疲れない提案リズムを保つ。

**体験目標**: 提案が負担にならない軽さ

### Quiet Satisfaction（静かな満足感）

成功時は過剰な祝賀ではなく、
静かな確認で十分。

**体験目標**: 「できた」という静かな満足

### Human Focus Priority（人間集中優先）

Runtime は人間の作業を最優先する。
常に人間のフローを尊重する。

**体験目標**: 人間の思考が主役である感覚

## 3. Experience Boundaries

| Situation | Experience Goal |
|-----------|----------------|
| idle runtime | invisible（見えない） |
| successful completion | quiet satisfaction（静かな満足） |
| continuation restore | natural continuity（自然な継続） |
| repeated suggestions | suppression（抑制） |
| restart recovery | reassurance（安心感） |
| human focus state | no interruption（中断なし） |
| failure / error | calm behavior（冷静な対応） |
| timeout | panic suppression（パニック抑制） |

---

## 4. Continuation Experience Design

### 原則: Soft Continuity（柔らかな継続性）

- continuation は「自然な再開」である
- 強制再開は行わない
- 人間の意図を尊重する再開
- 記憶の提示は最小限

**体験目標**: 昨日の続きとして自然に感じられる

### 禁止行為

| 禁止行為 | 理由 |
|---------|------|
| 自動再開 | 人間の意図を無視する |
| 強制提案 | フローを壊す |
| 冗長な文脈提示 | 認知負荷を増やす |
| 連続する3回の継続要求 | 人間を疲弊させる |

---

## 5. Memory Visibility Design

### 原則: Memory Transparency（記憶の透明性）

- memory は全部見せない
- 必要時のみ可視化
- summary を押し付けない
- archive を静かに扱う
- context を肥大表示しない

**体験目標**: 記憶が自然に存在している感覚

### 禁止行為

| 禁止行為 | 理由 |
|---------|------|
| memory の過剰提示 | 認知負荷を増やす |
| archive の自動表示 | フローを中断する |
| 冗長な要約 | 人間の集中を壊す |
| 失敗の記憶の押し付け | 不安を与える |

---

## 6. Notification Rhythm Design

### 原則: Interruption-Light（低中断）

- 通知頻度は最小限
- cooldown を設ける
- repeated interruption を防止
- silent mode を尊重

**体験目標**: フローを壊さない静けさ

### 禁止行為

| 禁止行為 | 理由 |
|---------|------|
| notification spam | フローを壊す |
| repeated interruption | 人間を疲弊させる |
| silent mode の無視 | 人間の選択を無視する |
| emergency 以外の常時通知 | 不安を与える |

---

## 7. Recovery Experience Design

### 原則: Recovery Reassurance（回復の安心感）

- restart 後の安心感
- session recovery の自然さ
- failure 後の calm behavior
- timeout 時の panic suppression

**体験目標**: 「また始まる」のではなく「また会える」感覚

### 禁止行為

| 禁止行為 | 理由 |
|---------|------|
| panic behavior | 不安を与える |
| エラーの過剰説明 | 認知負荷を増やす |
| 自動再試行 | 人間の意図を無視する |
| 冗長な recovery メッセージ | フローを壊す |

---

## 8. Ambient Runtime Presence

### 原則: Ambient Presence（環境的な存在感）

- idle 時は静か
- 必要時だけ foreground に上がる
- 常時アピール禁止
- human focus 優先

**体験目標**: 意識しなくても馴染む自然さ

### 禁止行為

| 禁止行為 | 理由 |
|---------|------|
| 常時アピール | フローを壊す |
| バッジ表示 | 注意を奪う |
| ポップアップ | 人間の集中を中断する |
| 自動通知 | フローを壊す |

---

## 9. Prohibited Behaviors（全体験共通の禁止行為）

以下の動作は厳禁：

| 禁止行為 | 理由 |
|---------|------|
| attention-seeking behavior | 人間の集中を壊す |
| excessive notifications | フローを中断する |
| aggressive continuation | 強制再開は不安を与える |
| forced summaries | 認知負荷を増やす |
| recursive UX optimization | 過剰設計で肥大化する |
| persistent interruption | 人間を疲弊させる |
